## 1. Executive Summary

에듀렌즈는 코딩 교육 환경에서 학생의 **학습 정체(Learning Stall)** 를 AI Vision으로 자동 감지하고, 소크라테스식 힌트로 자립적 문제 해결을 유도하는 플랫폼입니다. 학생에게는 AI 코파일럿을, 강사에게는 실시간 학습 현황 모니터링 대시보드를 제공합니다.

---

## 2. 문제 정의

### 배경

국내 부트캠프 및 온라인 코딩 교육 시장은 연간 수십만 명의 수강생이 참여하는 규모로 성장했습니다. 그러나 학습 효율 측면에서 두 가지 만성적인 문제가 존재합니다.

### 문제 1 — 학생: 침묵의 막힘 (Silent Stall)

학생이 코딩 과제나 실습 중 오류를 만났을 때:
- 질문하기 부끄러워 평균 27분간 혼자 씨름합니다.
- Stack Overflow나 ChatGPT에서 정답을 복사하여 붙여넣기하는 **"Copy-Paste 학습"** 패턴이 고착됩니다.
- 강사에게 도움을 요청해도 어디서 막혔는지 설명하는 데만 5~10분이 소요됩니다.

**결과**: 학습 시간 낭비, 낮은 자기효능감, 개념 이해 없는 진도 진행.

### 문제 2 — 강사: 사각지대 (Blind Spot)

온라인 또는 대규모 오프라인 강의에서 강사는:
- 어떤 학생이 어디서 막혀 있는지 **실시간으로 파악할 방법이 없습니다**.
- 학생이 "잘 모르겠어요"라고 말하기 전까지 상태를 알 수 없습니다.
- 수업 후 복기에 의존하여 피드백이 늦습니다.

**결과**: 강사 리소스 낭비, 개인화된 피드백 불가능, 위기 학생 조기 발견 실패.

## 3. 목표 사용자

### Primary User A — 학생 (Learner)
- **페르소나**: 부트캠프 또는 온라인 강의를 수강 중인 20~35세
- **기술 수준**: 입문~중급 (개발, 디자인, 데이터 분석 등 다양)
- **Pain Point**: 혼자 막혔을 때 어디서 도움을 받아야 할지 모름
- **Goal**: 빠른 힌트를 받되, 스스로 해결하는 성취감을 원함

### Primary User B — 강사 (Instructor)
- **페르소나**: 부트캠프 강사, 대학 조교, 온라인 강의 운영자
- **Pain Point**: 수강생 개개인의 실습 진행 상황을 실시간으로 모니터링하기 어려움
- **Goal**: 위기 학생(반복 실패)을 조기에 발견하고 개입 타이밍을 놓치지 않음

## 4. 핵심 가치 제안

| 가치 | 에듀렌즈의 접근 방식 |
|------|----------------------|
| **무인 감지** | 학생이 요청하지 않아도 AI가 화면에서 막힘을 자동 감지 |
| **소크라테스식 교육** | 정답 제공 대신 "왜 그럴까요?" 방식의 힌트로 자립적 학습 유도 |
| **도메인 전문성** | 카테고리별 전문가 AI 페르소나 (DevOps, 보안, 데이터 사이언스 등) |
| **강사 인사이트** | 어떤 학생이 어떤 유형의 문제에 반복적으로 막히는지 데이터화 |
| **제로 설치** | 별도 앱 설치 없이 브라우저에서 바로 실행 (Web `getDisplayMedia`) |

## 5. 유저 플로우

### 5.1 학생 플로우

```
회원가입 (Clerk)
    │
    ▼
온보딩 (닉네임 + 역할 설정)
    │
    ▼
학생 대시보드 (/)
    │
    ├─ [멘토 미설정] → MentorDialog (강사 검색 + 연결)
    │
    ▼
학습 카테고리 선택 (6개 중 택 1)
    │
    ▼
AI 코파일럿 위젯 활성화
    │
    ├─ "감지 시작" 클릭 → getDisplayMedia 권한 요청
    │       │
    │       ▼
    │  화면 공유 시작 → 30초 카운트다운
    │       │
    │       ▼ (30초마다 반복)
    │  스크린샷 캡처 → POST /api/analyze (auto)
    │       │
    │       ├─ 오류 감지됨 → 소크라테스 힌트 (Level 1 공개)
    │       │                  └─ "다음 힌트" 클릭 → Level 2, 3 순차 공개
    │       │
    │       └─ 정상 작업 → 선제적 질문 제안
    │                        └─ "예, 도와주세요" → 수동 Q&A 모드
    │                        └─ "아니오" → 자동 감지 일시 정지
    │
    ├─ 직접 질문 입력 → POST /api/analyze (manual) → AI 답변
    │
    └─ "세션 종료" → 세션 resolved 처리 → 학습 기록에 저장
```

### 5.2 강사 플로우

```
로그인 (Clerk)
    │
    ▼
관리자 대시보드 (/admin)
    │
    ├─ 요약 지표 확인 (담당 수강생 수, 활성 세션, 위급 상태)
    ├─ 차트 확인 (7일 상호작용 빈도, 유형 분포)
    │
    ▼
수강생 디렉토리
    │
    ├─ 에듀렌즈 사용 중 표시 (그린 닷)
    ├─ 상호작용 횟수 표시
    │
    └─ 카드 클릭 → 학생 타임라인 뷰
                    │
                    └─ 카테고리별 AI 응답 이력
                       (유형: 오류/선제적 조언/질의응답 + 시각)
```

## 6. 기능 요구사항

### F-01. 인증 및 역할 관리
- **F-01-1**: Clerk 기반 이메일/OAuth(Google, GitHub) 회원가입
- **F-01-2**: 온보딩 페이지에서 닉네임 + 역할(학생/강사) 설정
- **F-01-3**: JWT `publicMetadata`에 역할을 저장하여 미들웨어에서 역할 기반 라우팅
- **F-01-4**: 강사만 `/admin` 접근 허용; 학생은 `/` 접근

### F-02. 화면 감지 및 AI 분석
- **F-02-1**: 브라우저 `getDisplayMedia` API로 화면 공유 (별도 프로그램 불필요)
- **F-02-2**: 30초마다 JPEG 스크린샷 캡처 (quality: 0.85)
- **F-02-3**: 캡처 이미지를 base64로 인코딩하여 `/api/analyze`에 POST
- **F-02-4**: 오류 감지 모델 (temperature 0.2)과 대화 모델 (temperature 0.7)을 별도 운용
- **F-02-5**: 오류 유형 분류: `syntax`, `tool_usage`, `config`

### F-03. 소크라테스식 힌트 시스템
- **F-03-1**: 오류 감지 시 3단계 힌트를 순차적으로 공개
  - Level 1: 방향 힌트 한 문장
  - Level 2: 구체적 원인 + 도메인 지식 활용 해결 방법
  - Level 3: 심화 개념 + 다음 단계 안내
- **F-03-2**: 각 카테고리에 맞는 도메인 전문가 AI 페르소나 적용

### F-04. 선제적 AI 제안
- **F-04-1**: 오류가 없을 때 학생의 현재 작업을 파악하여 자연스러운 질문 제안
- **F-04-2**: "예, 도와주세요" → 수동 Q&A 모드 전환
- **F-04-3**: "아니오" → 자동 감지 일시 정지 (수동 질문은 계속 가능)

### F-05. 수동 Q&A 채팅
- **F-05-1**: 화면 캡처 컨텍스트를 포함한 자유로운 질문
- **F-05-2**: Enter 키 전송, Shift+Enter 줄바꿈

### F-06. 데이터 저장 및 로깅
- **F-06-1**: 모든 AI 상호작용(오류, 선제적 조언, 질의응답)을 `practice_logs`에 저장
- **F-06-2**: 오류 3회 이상 감지 시 세션 상태를 `critical`로 에스컬레이션

### F-07. 강사 대시보드
- **F-07-1**: `profiles` 테이블 기반 정확한 담당 수강생 목록 (세션 유무와 무관)
- **F-07-2**: 비활성 세션 제외한 활성/정체/위급 세션 수 표시
- **F-07-3**: 학생별 상호작용 타임라인 (카테고리, 유형, AI 응답, 시각 포함)
- **F-07-4**: 30초 폴링 기반 데이터 갱신 (Supabase RLS 우회)

## 7. 비기능 요구사항

| 항목 | 요구사항 |
|------|---------|
| **응답 시간** | AI 분석 응답 < 10초 (Gemini 2.5 Flash 기준) |
| **보안** | 서비스 롤 키, Gemini API 키는 서버 환경에서만 사용 |
| **확장성** | Vercel Serverless 배포로 트래픽에 따라 자동 스케일 |
| **브라우저 호환** | Chrome 107+, Edge 107+ (getDisplayMedia 지원 기준) |
| **SSR/CSR 분리** | 서버 데이터 패칭은 Server Component, 인터랙티브 UI만 Client Component |
| **타입 안전성** | TypeScript strict 모드, `as unknown as T` 패턴으로 서버 액션 타입 브릿지 |

## 8. 성공 지표 (Success Criteria)

### 핵심 사용자 지표
| 지표 | 목표 |
|------|------|
| 세션 당 평균 "막힘 감지" 횟수 | 측정 가능 (baseline 확립) |
| 학생의 힌트 Level 1만으로 해결 비율 | > 50% |
| 수동 Q&A vs 자동 감지 비율 | 측정 가능 |
| 강사의 대시보드 주간 사용 빈도 | > 3회/주 |

### 기술 지표
| 지표 | 목표 |
|------|------|
| AI 오류 오탐률 (false positive) | < 20% |
| API 응답 성공률 | > 99% |
| TypeScript 빌드 오류 | 0 |

## 9. 범위 외 항목 (Out of Scope)

- **모바일 앱**: 현재 버전은 데스크톱 브라우저 전용 (`getDisplayMedia` 제약)
- **다국어 지원**: 현재 한국어 전용
- **LMS 연동**: Canvas, Moodle 등 외부 LMS와의 직접 연동 없음
- **결제 시스템**: MVP 단계에서 제외
- **오프라인 지원**: 인터넷 연결 필수 (AI API 의존)
